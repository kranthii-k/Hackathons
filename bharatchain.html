Problem Statement

Why UPI fails — and why it matters

UPI processes over 500 million transactions daily, yet approximately 3% fail — leaving users in limbo, banks with reconciliation nightmares, and regulators with no reliable audit trail. The failures aren't random noise: they cluster around three precise failure modes, each with a different root cause and a different blockchain-native fix.

The critical insight: UPI's failure is not a network problem — it's a state agreement problem. Two parties transact but cannot agree on the final state. Blockchain provides exactly one thing UPI lacks: a shared, immutable ledger that all parties can trust without trusting each other.
⚡
Failure Type 01

Timeout Ambiguity

Money is debited from sender's account but the transaction times out before NPCI confirms credit. Both the bank and user are left with "pending" — no one knows if money moved or not. Resolution can take 3–5 business days through manual reconciliation.

⚙
Failure Type 02

Idempotency Violation

A network retry causes the same transaction to be processed twice at different bank nodes. UPI's current system lacks a cryptographically enforced idempotency key. Result: double debit, disputed with the PSP that may take weeks to resolve.

📋
Failure Type 03

Reconciliation Lag

NPCI and member banks settle net positions at T+1. In the 24-hour window, the ledger states of two banks can diverge — particularly during high-traffic events (festivals, salary days). Disputes arise from stale cached states.

System Architecture

BharatChain does not replace UPI — it inserts a permissioned blockchain middleware between the PSP gateway and member banks. Existing UPI apps work without modification. The blockchain layer is invisible to end users but provides cryptographic guarantees to all institutional participants.

// atomic_settle.go — BharatChain core chaincode
// Hyperledger Fabric 2.5 · Go 1.21

package main

import (
    "encoding/json"
    "fmt"
    "time"
    "crypto/sha256"
    "github.com/hyperledger/fabric-contract-api-go/contractapi"
)

// TxStatus represents the lifecycle state of a UPI transaction
type TxStatus string

const (
    StatusPending   TxStatus = "PENDING"
    StatusConfirmed TxStatus = "CONFIRMED"
    StatusFailed    TxStatus = "FAILED_SAFE"
    StatusDisputed  TxStatus = "DISPUTED"
    TimeoutWindow              = 30 * time.Second
)

// UPITransaction is the on-chain record — PII stored as ZK hash only
type UPITransaction struct {
    TxID           string   `json:"txId"`
    SenderHash     string   `json:"senderHash"`    // zk-SNARK hash of VPA
    ReceiverHash   string   `json:"receiverHash"`  // zk-SNARK hash of VPA
    AmountHash     string   `json:"amountHash"`    // Pedersen commitment
    SenderBank     string   `json:"senderBank"`
    ReceiverBank   string   `json:"receiverBank"`
    Status         TxStatus `json:"status"`
    InitiatedAt    int64    `json:"initiatedAt"`
    FinalizedAt    int64    `json:"finalizedAt"`
    IdempotencyKey string   `json:"idempotencyKey"` // prevents double-debit
    ProofHash      string   `json:"proofHash"`     // zk proof reference
}

type BharatChainContract struct {
    contractapi.Contract
}

// InitiateTx — called by NPCI hook when tx begins. Anchors hash on-chain.
func (bc *BharatChainContract) InitiateTx(
    ctx contractapi.TransactionContextInterface,
    txID, senderHash, receiverHash, amountHash,
    senderBank, receiverBank, idempotencyKey string,
) error {

    // Enforce idempotency — reject duplicate transaction keys
    existing, _ := ctx.GetStub().GetState("IDEM_" + idempotencyKey)
    if existing != nil {
        return fmt.Errorf("idempotency violation: tx %s already processed", idempotencyKey)
    }

    tx := UPITransaction{
        TxID:           txID,
        SenderHash:     senderHash,
        ReceiverHash:   receiverHash,
        AmountHash:     amountHash,
        SenderBank:     senderBank,
        ReceiverBank:   receiverBank,
        Status:         StatusPending,
        InitiatedAt:    time.Now().UnixMilli(),
        IdempotencyKey: idempotencyKey,
    }

    txBytes, _ := json.Marshal(tx)
    ctx.GetStub().PutState(txID, txBytes)
    ctx.GetStub().PutState("IDEM_"+idempotencyKey, []byte(txID))

    // Emit event for bank nodes to begin endorsement
    return ctx.GetStub().SetEvent("TxInitiated", txBytes)
}

// AtomicSettle — both banks must endorse. PBFT ensures 2f+1 agreement.
func (bc *BharatChainContract) AtomicSettle(
    ctx contractapi.TransactionContextInterface,
    txID, zkProofHash string,
    senderAck, receiverAck bool,
) error {

    txBytes, err := ctx.GetStub().GetState(txID)
    if err != nil || txBytes == nil {
        return fmt.Errorf("tx %s not found", txID)
    }

    var tx UPITransaction
    json.Unmarshal(txBytes, &tx)

    // Timeout check — if window expired, auto-fail-safe
    elapsed := time.Now().UnixMilli() - tx.InitiatedAt
    if elapsed > TimeoutWindow.Milliseconds() {
        tx.Status = StatusFailed
        txBytes, _ = json.Marshal(tx)
        ctx.GetStub().PutState(txID, txBytes)
        return ctx.GetStub().SetEvent("TxFailedSafe", txBytes)
    }

    // Atomic: both banks must ack. If one missing → disputed state.
    if senderAck && receiverAck {
        tx.Status = StatusConfirmed
        tx.FinalizedAt = time.Now().UnixMilli()
        tx.ProofHash = zkProofHash
    } else {
        tx.Status = StatusDisputed
    }

    txBytes, _ = json.Marshal(tx)
    ctx.GetStub().PutState(txID, txBytes)
    return ctx.GetStub().SetEvent("TxSettled", txBytes)
}

func main() {
    cc, _ := contractapi.NewChaincode(&BharatChainContract{})
    cc.Start()
}
// timeout_fsm.go — Finite State Machine for timeout resolution
// Resolves the hardest UPI failure: debit confirmed, credit unknown

package fsm

import (
    "context"
    "time"
    "log/slog"
)

type FSMState int

const (
    StateInit       FSMState = iota
    StateDebitSent            // sender bank confirms debit
    StateCreditPending        // awaiting receiver bank ACK
    StateConfirmed            // both banks confirmed
    StateFailSafe             // timeout → auto reverse debit
    StateDisputed             // conflicting signals → human review
)

type TimeoutFSM struct {
    TxID         string
    State        FSMState
    DebitAt      time.Time
    Window       time.Duration
    EvidenceCh   chan Evidence
    OnResolve    func(TxID string, finalState FSMState)
}

type Evidence struct {
    From    string   // "sender_bank" | "receiver_bank" | "npci"
    Signal  string   // "DEBIT_OK" | "CREDIT_OK" | "CREDIT_FAIL"
    At      time.Time
}

func NewTimeoutFSM(txID string, window time.Duration, onResolve func(string, FSMState)) *TimeoutFSM {
    return &TimeoutFSM{
        TxID:       txID,
        State:      StateInit,
        Window:     window,
        EvidenceCh: make(chan Evidence, 10),
        OnResolve:  onResolve,
    }
}

// Run — executes FSM. Blocks until resolved or timeout.
func (f *TimeoutFSM) Run(ctx context.Context) {
    timer := time.NewTimer(f.Window)
    defer timer.Stop()

    var (
        debitOK  = false
        creditOK = false
    )

    for {
        select {
        case ev := <-f.EvidenceCh:
            switch ev.Signal {
            case "DEBIT_OK":
                debitOK = true
                f.State = StateDebitSent
                slog.Info("debit confirmed", "tx", f.TxID, "from", ev.From)

            case "CREDIT_OK":
                creditOK = true
                if debitOK {
                    f.State = StateConfirmed
                    f.OnResolve(f.TxID, StateConfirmed)
                    return
                }

            case "CREDIT_FAIL":
                // Receiver bank explicitly rejected — safe to reverse
                f.State = StateFailSafe
                f.OnResolve(f.TxID, StateFailSafe)
                return
            }

            // Check full confirmation
            _ = creditOK

        case <-timer.C:
            // Timeout: if debit happened but no credit ACK → fail safe
            if debitOK && !creditOK {
                slog.Warn("timeout: debit-no-credit, triggering reversal", "tx", f.TxID)
                f.State = StateFailSafe
            } else if !debitOK {
                // Nothing moved — clean failure
                f.State = StateFailSafe
            } else {
                f.State = StateDisputed
            }
            f.OnResolve(f.TxID, f.State)
            return

        case <-ctx.Done():
            return
        }
    }
}
// npci_hook.js — gRPC middleware hook injected at NPCI PSP layer
// Intercepts UPI transactions and anchors them on BharatChain
// Runs as a sidecar alongside existing NPCI services

const grpc = require('@grpc/grpc-js')
const { Gateway, connect } = require('@hyperledger/fabric-gateway')
const crypto = require('crypto')
const { TextEncoder } = require('util')

const CHANNEL_NAME = 'bharatchain'
const CHAINCODE_NAME = 'atomic-settle'
const enc = new TextEncoder()

/**
 * BharatChainHook — wraps NPCI transaction calls
 * Call initiate() when UPI tx starts, settle() when banks respond
 */
class BharatChainHook {
    constructor(peerEndpoint, tlsCertPath, certPath, keyPath) {
        this.client = new grpc.Client(peerEndpoint, loadCredentials(tlsCertPath))
        this.identity = loadIdentity(certPath, keyPath)
        this.gateway = null
        this.contract = null
    }

    async connect() {
        this.gateway = connect({
            client: this.client,
            identity: this.identity,
            signer: createSigner(this.identity.credentials.privateKey),
        })
        const network = this.gateway.getNetwork(CHANNEL_NAME)
        this.contract = network.getContract(CHAINCODE_NAME)
        console.log('BharatChain hook connected ✓')
    }

    /**
     * initiateTx — anchor hash at transaction start
     * Called by NPCI PSP hook BEFORE forwarding to bank
     */
    async initiateTx(upiPayload) {
        const {
            txId, senderVPA, receiverVPA,
            amount, senderBank, receiverBank
        } = upiPayload

        // Hash PII — VPAs and amounts never stored in plaintext on-chain
        const senderHash = this.hashPII(senderVPA)
        const receiverHash = this.hashPII(receiverVPA)
        const amountHash = this.pedersenCommit(amount)
        const idempotencyKey = this.hashPII(`${txId}_${senderVPA}_${amount}`)

        try {
            await this.contract.submitTransaction(
                'InitiateTx',
                txId, senderHash, receiverHash,
                amountHash, senderBank, receiverBank, idempotencyKey
            )
            console.log(`[BC] TX ${txId} anchored on-chain`)
            return { success: true, idempotencyKey }
        } catch (err) {
            if (err.message.includes('idempotency violation')) {
                throw new Error(`DUPLICATE_TX: ${txId} already processed`)
            }
            throw err
        }
    }

    async settleTx(txId, senderAck, receiverAck, zkProofHash) {
        await this.contract.submitTransaction(
            'AtomicSettle', txId, zkProofHash,
            String(senderAck), String(receiverAck)
        )
        console.log(`[BC] TX ${txId} settled: sender=${senderAck} receiver=${receiverAck}`)
    }

    hashPII(value) {
        return crypto.createHash('sha256').update(value + process.env.BCRYPT_SALT).digest('hex')
    }

    pedersenCommit(amount) {
        // Simplified — use gnark Pedersen in production
        const r = crypto.randomBytes(32).toString('hex')
        return crypto.createHash('sha256').update(`${amount}:${r}`).digest('hex')
    }
}

module.exports = { BharatChainHook }
// zk_circuit.go — Zero-Knowledge proof circuit using gnark
// Proves: "A valid UPI transaction occurred between valid VPAs
//          for an amount within regulatory limits"
// WITHOUT revealing: VPAs, exact amount, bank identity

package zkupi

import (
    "github.com/consensys/gnark/frontend"
    "github.com/consensys/gnark/std/hash/mimc"
    "github.com/consensys/gnark/std/algebra/emulated/sw_bn254"
)

// UPIProofCircuit — defines the zk-SNARK circuit constraints
type UPIProofCircuit struct {
    // Private inputs (only the prover knows these)
    SenderVPA    frontend.Variable `gnark:",secret"`
    ReceiverVPA  frontend.Variable `gnark:",secret"`
    Amount       frontend.Variable `gnark:",secret"`
    Nonce        frontend.Variable `gnark:",secret"`

    // Public inputs (on-chain verifiable)
    SenderCommit  frontend.Variable `gnark:",public"`
    ReceiverCommit frontend.Variable `gnark:",public"`
    AmountRangeOK frontend.Variable `gnark:",public"` // 1 if 1 ≤ amt ≤ 1Cr
    TxHashPublic  frontend.Variable `gnark:",public"`
}

// Define — specifies circuit constraints. gnark compiles to R1CS.
func (c *UPIProofCircuit) Define(api frontend.API) error {
    // Constraint 1: SenderCommit = MiMC(SenderVPA, Nonce)
    h := mimc.NewMiMC(api)
    h.Write(c.SenderVPA, c.Nonce)
    api.AssertIsEqual(c.SenderCommit, h.Sum())

    // Constraint 2: ReceiverCommit = MiMC(ReceiverVPA, Nonce)
    h.Reset()
    h.Write(c.ReceiverVPA, c.Nonce)
    api.AssertIsEqual(c.ReceiverCommit, h.Sum())

    // Constraint 3: Amount in valid range [1, 100000000] (₹1 to ₹1Cr)
    // Uses range-check gadget — no need to reveal amount
    api.AssertIsLessOrEqual(c.Amount, 100_000_000)
    api.AssertIsLessOrEqual(1, c.Amount)
    api.AssertIsEqual(c.AmountRangeOK, 1)

    // Constraint 4: TxHash = MiMC(SenderVPA, ReceiverVPA, Amount, Nonce)
    h.Reset()
    h.Write(c.SenderVPA, c.ReceiverVPA, c.Amount, c.Nonce)
    api.AssertIsEqual(c.TxHashPublic, h.Sum())

    return nil
}

// ProveUPITx — generates a Groth16 proof for a given UPI transaction
func ProveUPITx(senderVPA, receiverVPA string, amount uint64) ([]byte, error) {
    witness := &UPIProofCircuit{
        SenderVPA:   stringToField(senderVPA),
        ReceiverVPA: stringToField(receiverVPA),
        Amount:      amount,
        Nonce:       randomField(),
    }
    // Compute public outputs from private inputs
    witness.computePublicOutputs()

    // Use pre-compiled proving key (generated in trusted setup phase)
    proof, err := groth16.Prove(compiledCircuit, provingKey, witness)
    if err != nil {
        return nil, fmt.Errorf("proof generation failed: %w", err)
    }

    var buf bytes.Buffer
    proof.WriteTo(&buf)
    return buf.Bytes(), nil
}

Privacy Design

Zero-knowledge privacy under DPDP Act 2023

Putting UPI transactions on a shared ledger sounds like a privacy nightmare. The ZK layer is the research answer. Using Groth16 zk-SNARKs (via gnark), BharatChain proves a valid transaction occurred — without ever storing VPAs, names, or exact amounts on-chain.

What goes on-chain (public)

  • MiMC hash of sender VPA (not reversible)
  • MiMC hash of receiver VPA (not reversible)
  • Pedersen commitment to amount (hides exact value)
  • Boolean: amount within legal range (₹1–₹1Cr)
  • Groth16 proof hash (verifiable in <5ms)
  • Bank identifiers (SBI → HDFC, not account numbers)
  • Timestamp, status, finality block number

What never leaves the NPCI layer

  • Actual VPAs (user@okaxis, etc.) — never on-chain
  • Account numbers or IFSC codes
  • Exact amount (only commitment + range proof)
  • User names, phone numbers, device IDs
  • Transaction notes / UPI remarks
  • IP addresses or location data
  • ZK secret nonce (stays with the prover)
DPDP Act 2023 compliance: The Digital Personal Data Protection Act treats VPAs and account numbers as personal data. By using ZK commitments, BharatChain's ledger contains no personal data — only mathematical proofs that a transaction was valid. RBI's audit node can verify settlement statistics without ever seeing individual transaction parties. This is the privacy model that makes regulator adoption possible.

Making It Real

How real users experience this

The entire point of BharatChain is that users never know it exists. They open PhonePe, send money, and it works — even in cases that previously failed. Behind the scenes, the system is fundamentally different.

Before BharatChain

  • User sends ₹500 to merchant. App shows "processing..." for 30 seconds
  • UPI times out. App shows "Transaction Failed" — but money is gone from sender's account
  • User calls bank. Bank says "raise a dispute." Dispute takes 5 business days
  • User has no way to verify if merchant received money or not
  • Manual reconciliation team at bank resolves it T+1 at best

After BharatChain

  • User sends ₹500 to merchant. App shows "processing..."
  • UPI times out at 30 seconds. BharatChain FSM detects: debit OK, credit unknown
  • Chaincode auto-triggers atomic reversal within 2 seconds of timeout
  • App shows "Transaction Failed — ₹500 returned to your account"
  • Entire resolution: under 35 seconds. Zero manual intervention needed

Institutional benefits

  • Banks reduce reconciliation staff costs by eliminating T+1 net settlement disputes
  • NPCI gets cryptographic audit trail without storing PII in new systems
  • RBI gets real-time aggregate settlement data via read-only node
  • PSPs get idempotency guarantees preventing double-debit class action risk

  • Apply: RBI Regulatory Sandbox (Payments theme) — open to research institutions

  • Compliance: Hyperledger Fabric permissioned network satisfies RBI cloud guidelines
  • Timeline: Sandbox approval ~6 months post-application; typically Cohort N+2

This solves domestic real-time retail settlement — a different problem at a different scale (500M/day vs millions/day). No published system addresses the UPI timeout ambiguity problem with a formal FSM and blockchain evidence model. That gap is the PhD contribution.
