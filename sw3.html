00 — Why This Problem

SW-3 is what we call a "trojan horse" problem — it looks deceptively simple on the surface ("an AI assistant") but the constraints make it genuinely hard and interesting: fully on-device, no network calls, adaptive learning, and transparent user control.

Most teams who attempt this will build a chatbot with a privacy banner slapped on it. The team that wins will build a system with actual behavior learning — a model that watches usage patterns, infers context, and proactively surfaces suggestions — all running locally. That's a fundamentally different thing and judges will notice.

The demo advantage is enormous. Unlike DICOM or RF classification, every judge personally understands what a good phone assistant feels like. When you show it correctly predicting "you usually open Spotify during your evening commute — want me to start it?" from purely local data, the reaction is visceral. That emotional response is what wins.

The digital wellbeing angle is the secret weapon. Build a screen-time awareness feature that nudges gently instead of nagging. Judges are humans with phones. They will feel this.

03 — 24-Hour Build Roadmap
Hour-by-Hour Execution Plan
0–2
Setup
Foundation Environment & Data Simulation
Your biggest challenge: you don't have real phone sensor data. Solve this upfront with a data simulator that generates realistic usage patterns.
Install: Python, Flask/FastAPI, SQLite, Ollama (pull gemma2:2b or phi3:mini — 1.6GB download, do this FIRST)
Build data_simulator.py: generates 30 days of fake app usage (Instagram, Spotify, Maps, Gmail, Camera) with realistic time distributions
Set up SQLite schema: usage_events, routines, suggestions, privacy_settings, audit_log tables
Decide platform: Python desktop app (easiest demo) OR Android (impressive but risky in 24h)
2–6
Core ML
Critical Routine Detection + Learning Engine
This is the heart of the demo. Build the system that actually learns — not fake-learns.
Routine detector: sliding 30-min windows on usage timestamps. If app X appears in same window 4+ times across 5+ days → routine detected. Store with confidence score.
Context signals: time of day (morning/afternoon/evening/night), day of week, session length, app sequence patterns (A → B within 5 mins)
Habit strength scorer: recency-weighted frequency. Recent events count more. Decays over 7 days. This prevents stale suggestions.
Drift detector: flag when a routine hasn't occurred in 3× its normal interval. Stop suggesting it.
6–10
LLM Layer
Critical On-Device LLM Integration
Connect the local LLM to your learned context. The trick is context injection — the LLM knows about you without you ever typing your history.
Run Ollama locally. Verify zero network calls with netstat (this is your privacy proof for judges)
Build context_injector.py: generates a system prompt from detected routines + current time + recent usage — injected automatically into every conversation
Build suggestion_engine.py: rule-based layer that fires proactive cards when context matches a detected routine within a time window
Implement intent classifier: is the user asking a question, making a request, or just chatting? Routes to appropriate handler.
10–15
UI / Demo
Demo Core The Interface Judges Will See
Split the screen into 3 panels — this layout is your demo story.
Panel 1 — Companion Chat: clean chat UI with streaming LLM responses. Context-aware opener: "Good evening! Based on your patterns, you usually wind down with Spotify around now — want me to shuffle your liked songs?"
Panel 2 — Live Insight Cards: real-time cards showing detected routines, today's suggestions, and app usage summary. Animate new suggestions sliding in.
Panel 3 — Privacy Dashboard: everything the system knows, visualized. Per-app toggle switches. "Forget this routine" buttons. A live network monitor showing 0 outbound calls.
Add Wellbeing Nudge: simulate opening Instagram 8 times in 20 minutes → gentle card appears: "Looks like you've checked Instagram 8 times this hour. Want a 20-min break?"
15–19
Privacy
Privacy Transparency & Control System
This is what separates you from "just another AI assistant." Make privacy tangible and demonstrable, not just a claim.
"What I Know About You" full report: shows every routine, every app stat, every inference — with timestamps and confidence scores. Download as JSON.
Granular reset: reset a single app's data, reset a date range, or full wipe. Each action writes to audit_log with timestamp.
Learning pause: per-app toggle that stops tracking immediately. Show the routine detector ignoring events from disabled apps in real-time.
Data expiry engine: slider from 7–90 days. Old events auto-deleted. Show storage size dropping as you move slider.

04 — Key Code Skeletons
Core Implementation Patterns
routine_detector.py
# Detects recurring daily routines from app usage timestamps import sqlite3, pandas as pd from collections import defaultdict from datetime import datetime, timedelta def detect_routines(db_path, min_occurrences=4, window_mins=30): conn = sqlite3.connect(db_path) df = pd.read_sql("SELECT app_name, timestamp FROM usage_events", conn) df['dt'] = pd.to_datetime(df['timestamp']) df['time_bucket'] = (df['dt'].dt.hour * 60 + df['dt'].dt.minute) // window_mins df['day_of_week'] = df['dt'].dt.dayofweek routines = [] for (app, bucket, dow), group in df.groupby(['app_name', 'time_bucket', 'day_of_week']): if len(group) >= min_occurrences: # Recency-weighted confidence days_ago = (datetime.now() - group['dt'].max()).days confidence = len(group) / (days_ago + 1) * 0.8 routines.append({ "app": app, "time_window": f"{bucket * window_mins // 60:02d}:{bucket * window_mins % 60:02d}", "day": ["Mon","Tue","Wed","Thu","Fri","Sat","Sun"][dow], "occurrences": len(group), "confidence": round(min(confidence, 1.0), 2), }) return sorted(routines, key=lambda x: x["confidence"], reverse=True)
context_injector.py — LLM system prompt builder
import ollama from datetime import datetime def build_context_prompt(routines, recent_apps, db): now = datetime.now() time_of_day = ["night","morning","afternoon","evening"][now.hour // 6] top_routines = "\n".join( f"- Uses {r['app']} around {r['time_window']} on {r['day']}s ({r['confidence']*100:.0f}% confidence)" for r in routines[:3]) return f"""You are a private AI companion running entirely on this device. You have NO internet access and make ZERO network calls. WHAT YOU KNOW ABOUT THE USER (learned locally): - Current time: {now.strftime('%H:%M')}, {time_of_day} on a {now.strftime('%A')} - Recently used apps: {', '.join(recent_apps[-5:])} - Detected routines: {top_routines} RULES: - Never ask for or store sensitive data - Only suggest things relevant to detected patterns - If uncertain, say so honestly rather than guessing - Be helpful, concise, and non-intrusive""" def chat(user_message, history, routines, recent_apps, db): system = build_context_prompt(routines, recent_apps, db) response = ollama.chat( model="gemma2:2b", # or "phi3:mini" messages=[{"role":"system", "content": system}] + history + [{"role":"user", "content": user_message}], stream=True # streaming for responsive UI ) for chunk in response: yield chunk['message']['content']
wellbeing_guard.py — Digital wellbeing detection
from collections import deque from datetime import datetime, timedelta class WellbeingGuard: def __init__(self): self.recent = deque(maxlen=50) # last 50 events self.nudge_cooldown = {} # avoid spamming nudges def record(self, app_name): self.recent.append((app_name, datetime.now())) def check_compulsive(self, app_name, window_mins=20, threshold=6): cutoff = datetime.now() - timedelta(minutes=window_mins) count = sum(1 for a, t in self.recent if a == app_name and t > cutoff) if count >= threshold: last = self.nudge_cooldown.get(app_name, datetime.min) if (datetime.now() - last).seconds > 1800: # 30min cooldown self.nudge_cooldown[app_name] = datetime.now() return { "type": "gentle_nudge", "message": f"You've opened {app_name} {count}× in {window_mins} mins. " f"A short break might help you recharge 🌿", "severity": "low" # never alarming, always gentle } return None

07 — Avoid These Mistakes
Common Failure Modes
⚠️
Trying to Build for Real Android
Android development + on-device ML + custom DB + privacy controls = 72+ hours of work minimum. You have 24.
Build a desktop simulation. Pre-load realistic 30-day usage data. The UX story is identical. Judges care about the concept, not the platform.
⚠️
Using a Cloud LLM "Just for Demo"
Making even one API call to OpenAI/Gemini API instantly invalidates your entire privacy claim. Judges will ask directly.
Use Ollama exclusively. Pull the model in Hour 0. Test it works offline by turning off WiFi before your demo — then leave WiFi off during the demo.
⚠️
Fake "Learning" That's Just Hardcoded
Hardcoding suggestions like "You usually use Spotify at 7pm" without actual pattern detection is immediately obvious to technical judges.
Build real detection. The routine_detector.py above is genuinely ~60 lines. Run it on your simulated data and show the output matches what a real user would see.
⚠️
Neglecting the Privacy UX
Saying "it's private" is worthless. Showing a privacy dashboard where data can be viewed and deleted is credible. This is 30% of the problem statement.
The transparency dashboard must be a first-class feature, not an afterthought. Pre-assign one team member to own this exclusively from Hour 0.
⚠️
LLM That Ignores Context
If your LLM doesn't know anything about the user's patterns and just answers generic questions, you've built ChatGPT offline — not an adaptive companion.
The context_injector is non-negotiable. Every single LLM call must include the user's detected routines and recent app usage in the system prompt.
⚠️
Slow LLM Response Killing the Demo
On CPU, a 7B model can take 30s per response. Judges lose interest after 10 seconds of silence.
Use Gemma-2B or Phi-3-mini (not 7B). Enable streaming so text appears token-by-token. Prep a GPU laptop if available. The UI should show a typing indicator immediately.
